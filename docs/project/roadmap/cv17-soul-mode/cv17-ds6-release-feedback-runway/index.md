# CV17.DS6 — Pre-release Usage Adjustments, Packaging, And Feedback Runway

**Status:** ✅ Done

**Placement:** CV17 first-release closeout story

**User-visible outcome:** The first Soul Mode release is tuned from real Pi usage, packaged, documented, validated, and explicitly leaves advanced voices and richer curation for feedback-driven expansion.

---

## Why This Exists

Soul Mode has a large product horizon: Wisdom Voice, Beauty Voice, Passagem, Return To Center, poetry, literature, philosophical fragments, richer ritual surfaces, and possible web support. The first release should not pretend to finish that ecosystem. It should close the validated minimal grammar and leave the rest as a conscious runway.

---

## Scope

### Pre-release usage adjustment pass

- Validate the full ritual in Pi through natural language.
- Tune prompts, voice behavior, thresholds, and microcopy from real usage.
- Fix command-skill contract leaks where Pi fails to render required surfaces.
- Adjust fruit/harvest timing so the grammar feels natural rather than mechanical.
- Keep changes bounded to first-release coherence, not new feature expansion.

### Packaging and feedback runway

- Update public and internal docs for the first Soul Mode release.
- Add release notes when the version target is known.
- Run automated validation and a manual smoke route for Soul Mode.
- Document known limitations and deferred voices or rites.
- Update roadmap status across CV17 and completed stories.
- Add worklog evidence and validation notes.
- Preserve feedback prompts for future Soul Mode expansion.
- Record the post-release runway for a Closing Rite that reviews harvested material for psyche-layer enrichment proposals.

---

## Non-goals

- No new behavior beyond pre-release usage adjustments and closeout fixes needed for coherence.
- No implementation of full Wisdom Voice, Beauty Voice, Passagem, or Return To Center.
- No expansion into web UI unless a previous story made it necessary.
- No release packaging before the Navigator validates the core flow.

---

## Acceptance Behavior

Given DS1 through DS5 are complete, DS6 first absorbs real-use adjustments, then prepares a coherent release package with docs, validation route, and known limitations.

Given pre-release adjustment is complete, the full ritual can be completed in Pi without the Navigator needing to explain the grammar.

Given the first release is packaged, the roadmap clearly distinguishes shipped first-slice behavior from deferred Soul Mode expansion.

Given the Navigator wants to continue Soul Mode after the first release, the remaining work is visible as feedback-driven future scope rather than hidden unfinished work.

Given future Soul Mode integration is documented, the roadmap distinguishes the first-release journal harvest from a later Closing Rite that may propose updates to Self, Shadow, Ego behavior, personas, or journey memory only with explicit confirmation.

## Future Runway: Closing Rite And Psyche Enrichment

The first release closes with Harvested Fruit and optional journal persistence. A future Soul Mode expansion should add a Closing Rite that asks what wants to remain in the psyche, not only what should be recorded in the journal.

Potential future CV:

```text
CV18 — Soul Mode Integration And Psyche Enrichment
```

Candidate delivery stories:

```text
CV18.DS1 — Closing Rite And Integration Review
CV18.DS2 — Self Layer Enrichment Proposal
CV18.DS3 — Shadow Layer Enrichment Proposal
CV18.DS4 — Ego Behavior And Persona Integration
CV18.DS5 — Confirmation And Safe Identity Mutation
```

The Closing Rite should review harvested material and propose, never silently apply, possible enrichments to psyche layers:

- `self/soul`: principles, postures, refusals, and practices being incorporated;
- `shadow/*`: recurring rejected parts, protections, fears, compulsions, or disowned needs;
- `ego/behavior`: operational conduct the Mirror should embody;
- personas: specialized lenses refined by repeated material;
- journey memory: durable insights that belong to a specific journey.

Shadow enrichment requires special care. The Shadow layer must be treated as recurring listening hypotheses, not diagnosis or fixed identity. Mirror should never accuse the user of being a shadow pattern. It should ask what the rejected part protects and whether that pattern deserves to be remembered as a future listening lens.

Rule of gold:

```text
No automatic psyche mutation.
```

Soul Mode may propose:

```text
this may belong to Shadow
this may refine Self
this may update Ego behavior
```

But identity changes require an explicit confirmation flow with visible diff before mutation.
