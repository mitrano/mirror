# v0.29.3 — Pi External-Skill Discovery Fix

**Date:** 2026-07-04

## Highlights

- Fixes Pi external-skill discovery on the `~/.mirror-minds` home layout: the Pi logger extension now resolves the mirror home the same way the Python core does, instead of hardcoding the legacy `~/.mirror` root.
- Replicates the core's resolution contract in the extension — shell env > `.env` (upward walk, same parse rules as `config.py`) > single-catalog auto-scan — preferring the new `~/.mirror-minds` home and falling back to legacy `~/.mirror` only when the new one is absent.
- Returns null on ambiguity instead of guessing, so a multi-home machine requires an explicit `MIRROR_USER`/`MIRROR_HOME` rather than silently picking the wrong home.
- Records the resulting Python/TypeScript resolution duplication as accepted, tracked technical debt (TD-001) with an explicit revisit trigger.
- Hardens the Gemini smoke test's production-DB guard to resolve the DB path through the core so it protects the database the workspace actually uses, degrading to a no-op if resolution fails.

## Where We Started

`v0.29.2` closed the Ariad Builder flow refinement arc. Dogfooding on a machine
using the current `~/.mirror-minds` home layout then surfaced a quiet but real
regression: Pi reported **zero** installed external skills even when extensions
were installed.

The cause was a divergence between two implementations of one contract. The
Python core resolves the mirror home through `resolve_mirror_home` — honoring
shell environment, a per-workspace `.env`, `MIRROR_HOME`/`MIRROR_USER`, and the
`~/.mirror-minds` default with legacy `~/.mirror` fallback. The Pi logger
extension, written in TypeScript and unable to import `memory.config`, instead
hardcoded the legacy `~/.mirror` root and never loaded the workspace `.env`. On
`~/.mirror-minds` machines it read a non-existent catalog and, because runtime
extensions are designed to fail quietly, exposed no external skills with no
visible error.

## What Changed

The Pi logger extension now replicates the core's mirror-home resolution
contract: shell environment first, then an upward `.env` walk using the same
parse rules as `config.py`, then a single-catalog auto-scan. It prefers the new
`~/.mirror-minds` home and falls back to legacy `~/.mirror` only when the new one
is absent. On an ambiguous multi-home machine it returns null instead of
guessing, forcing an explicit `MIRROR_USER`/`MIRROR_HOME` rather than picking the
wrong home. The home-directory names are centralized as constants and
`_resolveMemoryDir`'s fallback is aligned with `config.DEFAULT_MEMORY_DIR`.

Because a Pi (TypeScript) extension cannot import the Python `memory.config`
constants, the fix necessarily duplicates the resolution contract across two
languages. That duplication is recorded as **TD-001** in the technical-debt
ledger with an explicit revisit trigger: any change to the core's mirror-home
resolution semantics (new precedence, a new `MIRROR_*` variable, or another
home-directory rename), or the next time a non-Python runtime surface needs to
resolve the home, should promote the resolution to a single source of truth
rather than adding a third copy.

The same mirror-home audit surfaced a parallel defect in the Gemini smoke test.
Its production-DB guard hardcoded `~/.mirror/<user>/memory.db` to checksum the
"real" production database before and after the smoke run. On `~/.mirror-minds`
machines that path is a stale legacy database or absent, so the guard silently
checksummed the wrong database — or nothing — leaving the in-use production
database unprotected while reporting success. The guard now resolves the path
through the core (`resolve_mirror_home` + `default_db_path_for_home`), runs before
the `DB_PATH`/`MEMORY_ENV` exports, and degrades to a no-op if resolution fails.

## Conscious Exclusions

- This release does not consolidate the duplicated mirror-home resolution into a single cross-language source of truth. That is deliberately deferred and tracked as TD-001; this patch centralizes the contract on each side instead.
- No new user-facing features ship in this release. It is a correctness and test-hardening maintenance patch.
- Push, tag creation, stable promotion, and GitHub Release publication remain explicit release operations, not side effects of the fix.

## What We Learned

Fail-quietly runtime extensions trade visibility for session safety, and that
trade has a cost: a home-resolution divergence can expose zero external skills
for a long time with no signal. Duplicated contracts across languages are a
standing drift risk — the right response here was to make the duplication
correct and explicitly tracked, not to pretend it does not exist.

## Next Horizon

The tracked follow-up is TD-001: when the core's home-resolution semantics next
change, or another non-Python runtime surface needs to resolve the home, promote
the resolution to a single source of truth — a core-exposed resolved-home command
or a language-neutral config the extension can call — rather than maintaining a
third copy, and add a regression check asserting Node and Python agree for
representative env/`.env` cases.
