# v0.17.1 — Builder Project-Path Guard

**Date:** 2026-05-28

## Highlights

- Fixes `memory build load <slug>` so the clone-role guard evaluates the journey `project_path`.
- Allows Builder Mode from a production Mirror runtime when the journey project itself is a `dev` clone.
- Keeps the conservative fallback to the current directory when no `project_path` is configured.
- Updates runtime boundary docs and decisions to describe the journey-scoped guard.

## Where We Started

The clone-role guard was introduced to prevent accidental development inside the
Mirror production checkout. That safety boundary was correct, but its placement
was too broad: Builder Mode inspected the current runtime directory before it had
loaded the journey. If the user ran a production Mirror runtime and selected a
journey whose project lived elsewhere, Builder could refuse before seeing the
project path that actually mattered.

The result was friction in the wrong place. The guard protected the runtime clone
rather than the project clone that Builder was about to edit.

## What Changed

Builder Mode now loads the journey first, reads its configured `project_path`,
and passes that path into clone-role inspection. If that project checkout is
marked `dev`, Builder proceeds even when the Mirror runtime itself is running
from a production clone. If the project checkout is marked `production`, Builder
still refuses unless `--ignore-production-role` is passed.

For journeys without a configured `project_path`, the command keeps the prior
conservative behavior and inspects the current directory. That keeps unscoped
Builder sessions safe by default.

## Conscious Exclusions

This patch does not change runtime update behavior, clone-role marker semantics,
or the override flag. It also does not add automatic project-path discovery; the
journey must still carry the correct `project_path` metadata for the guard to be
project-scoped.

## Next Horizon

Production should update normally to v0.17.1, then validate a Builder activation
from the production Mirror runtime against a journey whose project path points to
a development checkout.
