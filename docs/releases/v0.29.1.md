# v0.29.1 — Windows Compatibility Hardening

**Date:** 2026-06-25

## Highlights

- Renames committed Claude skill directories from `mm:*` to Windows-safe `mm-*` while preserving `name: "mm:*"` in each `SKILL.md`.
- Generates the canonical Claude plugin with Windows-safe skill directories while preserving the same Claude command names in skill frontmatter.
- Adds Windows-safe external extension skill materialization, including opportunistic cleanup from legacy `ext:*` directories to `ext-*` directories when the legacy directory belongs to Mirror.
- Prefers UTF-8 stdout and stderr at the `python -m memory` entrypoint so Unicode runtime surfaces remain printable on Windows consoles.
- Adds `python -m memory repair-encoding` as an explicit dry-run/apply command for reversible UTF-8/Windows mojibake in user text, with backup before mutation.
- Normalizes path display in runtime, Builder, Explorer handoff, web command, and web preference surfaces so rendered contracts use forward-slash paths consistently.

## Where We Started

A Windows local installation exposed two compatibility gaps. The repository still carried Claude skill directories such as `.claude/skills/mm:build`, which are valid on macOS and Linux but invalid on NTFS. The runtime also surfaced historical Portuguese text that had been stored as mojibake, for example `Ã©`, `Ã§`, and `Ã£`, after older Windows/local-runtime combinations decoded UTF-8 through legacy code pages.

The first risk was structural: solving Windows by deleting Claude skills would make the checkout portable but weaken the Claude runtime surface for everyone. The second risk was data-related: repairing mojibake automatically during startup or update would mutate user-owned memory without review.

## What Changed

Mirror now separates command identity from filesystem identity. Claude command names remain `mm:*` in `SKILL.md` frontmatter and user-facing documentation, but the committed directories use Windows-safe `mm-*` names. The same rule applies to the canonical Claude plugin generated under `plugins/mirror-mind/skills/`.

External extension materialization now uses filesystem-safe directories too. A Claude extension command such as `ext:review-copy` is stored under `ext-review-copy` on disk while the catalog continues to record the runtime command name. Install, expose, and uninstall paths know how to remove legacy `ext:*` directories when the legacy `SKILL.md` declares the same command or safe label, avoiding duplicate stale surfaces without deleting unrelated user-authored directories.

The runtime entrypoint now prefers UTF-8 output streams when possible. This keeps Mirror's Unicode cards, symbols, and Portuguese text printable in Windows console contexts while leaving capture streams and already-UTF-8 environments alone.

The new `repair-encoding` command handles historical database text explicitly. It scans known user-text columns, previews each reversible repair by default, and applies changes only with `--apply`. The apply path creates a database backup first unless the user explicitly opts out with `--no-backup`.

Several rendered path fields now use forward-slash display consistently. This keeps Ariad and web/runtime surfaces stable across platforms while leaving real filesystem operations under `Path` control.

## Conscious Exclusions

- No automatic database text repair runs during startup, diagnosis, or runtime update.
- No attempt is made to repair irrecoverable replacement-character corruption or arbitrary extension tables.
- No user-facing Claude command names were changed; `/mm:build`, `/mm:mirror`, and the rest remain the semantic command surface.
- No hosted or cross-device Windows synchronization behavior was added.
- No broader TypeScript port or runtime packaging change is included in this patch.

## What We Learned

Windows compatibility is not only an operating-system detail. It forces a clearer boundary between semantic names and filesystem names. Command names can keep the language users and runtimes understand, while directories use a portable representation.

Data repair needs a different posture from runtime hardening. Encoding defaults can be improved automatically because they affect future output. Historical user text belongs to the user, so repair must remain visible, previewable, and backup-gated.

## Next Horizon

The next compatibility horizon is release-channel validation: stable updates should carry the Windows-safe skill layout and repair command through the same self-update path real users run. Longer term, CV21 and CV22 continue the runtime packaging and TypeScript-port work that will reduce platform-specific seams further.
