# v0.21.1 — Mirror Mode Conversation Continuity

**Date:** 2026-06-03

## Highlights

- Repeated Mirror Mode context loads no longer split an active conversation.
- Persona/journey context is bound to the current open runtime conversation.
- Explicit conversation switching remains available through `conversation-logger switch` / new-conversation flows.
- Regression coverage protects Pi sessions that call `mirror load` repeatedly while processing one long conversation.

## Where We Started

During a `vida-economica` bill-payment session on 2026-06-01, a single Pi session was recorded as many short conversations. Investigation showed that repeated `uv run python -m memory mirror load --persona ... --journey ...` calls were being treated as conversation boundaries.

That was wrong. Loading Mirror context is not the same as starting a new conversation.

## What Changed

Mirror Mode now uses a binding path for normal context loads. If the runtime session already has an open conversation, the resolved persona and journey are written onto that conversation and runtime session. A new conversation is created only when the session does not already have an open conversation.

The existing explicit switch path remains intact for user-invoked new-conversation boundaries.

## Conscious Exclusions

- This release does not merge historical split conversations automatically.
- It does not change `mm-new` or explicit conversation switching semantics.
- It does not alter extraction, title generation, or journey inference policies.

## What We Learned

Context activation and conversation boundaries must stay separate. A command can need to refresh identity, persona, journey, and attachments many times inside one runtime session without implying a new memory record.

## Next Horizon

The repaired `vida-economica` session should be treated as the audit case for this class of bug. Future runtime integration work should preserve the same invariant: repeated context loads are idempotent with respect to conversation identity.
